# App Patterns

Use this file for onboarding, patient or member portals, dashboards, progress views, check-ins, and health-related forms.

## Product posture

- Make the product feel guided and supervised, not self-serve and risky.
- Present the next best action clearly on each screen.
- Use status and progress components to reduce uncertainty.

## Onboarding

- Start with intent and reassurance: what the setup does, how long it takes, and why the information matters.
- Use progressive disclosure. Break long forms into short themed steps.
- Explain sensitive fields in plain language.
- Provide a clear progress indicator and the option to save or resume when appropriate.

## Dashboard

- Lead with the current plan, next task, or practitioner-relevant update.
- Show a small number of prioritized modules instead of many equal-weight cards.
- Mix dense and light sections to avoid the “grid of sameness” problem.
- Prefer timeline, checklist, and state summaries over noisy analytic widgets.

## Portal and record views

- Make history legible with dates, status labels, and concise summaries.
- Show what changed, what needs review, and what happens next.
- Use calm empty states that suggest the next useful action.

## Form design

- Use one column by default.
- Group fields by mental model, not database schema.
- Show inline guidance before errors when possible.
- Ask only what is needed for the current step.

## Reassurance patterns

- Privacy note near sensitive inputs
- Contact path to clinician, coach, or support
- Clear response-time expectations
- Explanations for unusual terminology
- Confirmation state with next-step clarity

## Dashboard modules that work well here

- Today’s plan
- Program milestones
- Message or review status
- Symptom or check-in summary
- Upcoming appointment or follow-up
- Adherence streak, when it supports the experience without shame
